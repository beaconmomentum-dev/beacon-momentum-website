01 · OVERVIEW
The Phoenix Multi-Agent Architecture

Phoenix is the autonomous AI operating system for the Beacon Momentum ecosystem. It is not a single chatbot or a simple automation script. It is a structured organization of specialized AI agents — each with a defined role, defined authority, and defined limits — designed to run the day-to-day operations of a multi-brand digital enterprise with minimal human intervention.

The architecture is modeled on the same organizational principles that govern any well-run enterprise: a visionary at the top who sets direction, a director who translates vision into operational tasks, domain managers who own functional areas, and a swarm of specialist workers who execute. A dedicated maintenance agent watches the entire fleet and self-heals when components fail.

03 · AGENT ROSTER
Every Agent, Defined

Each agent has a single domain of authority. They cannot exceed it. The Trust Ladder (Section 05) governs how much autonomy each agent is granted over time.

BOB — THE VISIONARYTIER 0Human Principal
The only human in the chain. Communicates exclusively with Phoenix via Telegram or voice memo. Sets strategic direction. Approves all financial actions above the defined threshold. Never manages individual agents directly.
PHOENIX — THE DIRECTORTIER 1
Chief of Staff. The only agent with cross-container network visibility. Receives commands from Bob, breaks them into projects, delegates to domain managers, monitors quality, and escalates to Bob only when a decision requires human authority. Runs on Claude Opus.
Reports to → Bob (human)
BOSUNTIER 2
Owns all social content and community management across all four brands. Posts on a defined cron schedule.
Reports to → Phoenix
PICKETTIER 2
Owns all paid media operations — Meta Ads, campaign builds, creative uploads, budget monitoring, and performance reporting.
Reports to → Phoenix
HELMTIER 2
Data, reporting, and intelligence. Reads all platform analytics, generates daily briefings, and detects anomalies. Read-only — never writes, posts, or spends.
Reports to → Phoenix
THE ARCHITECTTIER 2
Owns all build, code, and GoHighLevel CRM operations. Responsible for funnel deployments, landing page builds, and automation workflows.
Reports to → Phoenix
THE STEWARDTIER 2
Capital and financial oversight. Never moves capital without explicit Bob approval — every financial action requires a human trigger.
Reports to → Phoenix
THE ALCHEMISTTIER 2 SPECIALIST
PhD-level DeFi analyst. Monitors protocol mechanics, yield strategies, liquidity pool dynamics, on-chain risk signals, governance proposals, and smart money wallet movements. Synthesizes findings into plain-English briefings with risk ratings.
⚠ Analyst only. Does not execute trades. Does not connect to any wallet. Does not move capital. Every output is a recommendation, never an action.
Reports to → The Steward
THE MEDICCROSS-TIER
Runs continuously in the background. Pings every active agent on a cron schedule. Monitors all container logs for failed tasks, crashed processes, or API errors. Attempts automated recovery before escalating.
⚠ Read access to all container logs. Write access only to its own repair queue. Cannot modify any other container's code or configuration.
Reports to → Phoenix (escalation only)
TIER 3 — THE SWARM (DOERS)

Ten specialist Doer agents, each performing exactly one task type: Copy, Image, Creative, Budget, Analytics, Alerts, GHL Builder, Code, Grant, and Report. Doers receive a task, complete it, attach a deliverable, and shut down. They hold no persistent context. They run on Tier 3 local models at zero token cost.

04 · INFRASTRUCTURE
Container Architecture & Hardware

The current implementation uses Docker Compose for container orchestration. The architecture is designed to be Kubernetes-ready — clean container boundaries, single-responsibility services, and stateless Doers that can be horizontally scaled without refactoring.

Mac Studio (Local)
Hosts Bosun, Picket, and Tier 3 Doer agents running local open-source models (Qwen, Llama 3). Apple Silicon unified memory architecture allows multiple models to run simultaneously. Token cost for Tier 3 workloads: $0.00.
DigitalOcean Droplet (Always-On)
Hosts Phoenix, Helm, and the Medic. Runs 24/7 independent of the Mac Studio's sleep state. Docker Compose auto-restarts all containers on reboot. This is the always-on backbone of the fleet.

Note on Kubernetes Migration Path

The current Docker Compose implementation is intentionally designed for clean Kubernetes migration. When the fleet reaches sustained capacity above 80%, migration to a managed Kubernetes cluster (GKE or EKS) is the defined next step. Container boundaries are already clean, services are stateless where possible, and the Doer pool is designed for horizontal scaling.

05 · TRUST LADDER
How Autonomy Is Earned

No agent is granted full autonomy on day one. Every agent begins at Step 1 and advances only after demonstrating reliable, accurate behavior at the current step. Financial actions are permanently gated at Step 4 — human approval is always required.

01
Read-Only Access
Agent can observe and report. Cannot write, post, or spend.
02
Draft & Approve
Agent produces output for human review before publishing.
03
Auto-Execute (Low Risk)
Agent executes pre-approved action types autonomously.
04
Full Autonomy
Agent operates independently within defined domain boundaries.
⚠
NEVER — Financial w/o Approval
Capital movement always requires explicit Bob authorization.
